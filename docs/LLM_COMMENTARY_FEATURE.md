# LLMコメンタリー機能 計画ドキュメント

## 📋 概要

ローカルLLMを導入し、その日にあったできごとを読んで感想を書く機能を追加する。
システムプロンプトで設定したパーソナリティがイベントに対して独自の視点でコメントを生成する。

---

- [ ] ボットに個性を持たせ、フォロワーとのエンゲージメント向上

### 期待される効果

- [ ] イベント間の関連性や歴史的文脈の提供
- [ ] フォロワーとの対話のきっかけ創出

## 🔍 要件整理

### 1. ローカルLLM選定

#### 検討候補
| LLM実行環境 | メリット | デメリット | 推奨度 |
|------------|---------|-----------|--------|
| **Ollama** | セットアップ簡単、モデル管理が容易、API互換性高い | - | ⭐⭐⭐⭐⭐ |
| **llama.cpp** | 軽量、高速 | 手動セットアップ必要 | ⭐⭐⭐ |
| **vLLM** | 高速推論 | メモリ消費大 | ⭐⭐ |
| **LocalAI** | 多機能 | 複雑 | ⭐⭐ |

#### 選定モデル（Raspberry Pi 5, 8GB RAM想定）
- **第一候補**: `qwen3:1.7b-instruct-q4_K_M` - 最新モデル（2025年4月29日リリース）
- **フォールバック**: `qwen3:1.7b-instruct-q3_K_M` - さらに軽量（第一候補が重い場合）

**Qwen3の利点**:
- 2025年4月29日リリースの最新モデル
- Qwen2.5から大幅に改善された性能
- 1.7Bと軽量ながら高品質な日本語生成
- 低メモリ消費（q4_K_Mで約1.2GB、q3_K_Mで約0.9GB）
- より自然で個性的な文章生成が可能

### 3. 機能仕様

#### 基本フロー（v1.0 - シンプル実装）

```
1. 定期投稿トリガー（00:00） ※現在は1日1回のみ
   ↓
2. その日のイベントを取得（既存機能）
   ↓
3. 通常の年表投稿を実行（既存の動作）
   ↓
4. 【5分待機】
   ↓
5. 投稿内容（イベントテキスト）をLLMに送信
   ↓
6. LLMが感想コメントを生成（100トークン程度）
   ↓
7. 元の投稿へのリプライとして感想を投稿（00:05頃）
```

**確定仕様**:
- 投稿時刻: 元投稿（00:00）の5分後 → **00:05**
- 投稿頻度: **1日1回のみ**（現在の定期投稿が1日1回のため）
- 既存の定期投稿機能には一切手を入れない
- LLM機能は独立した追加機能として実装
- 失敗しても既存の投稿には影響なし

#### 投稿フォーマット（v1.0 決定版）

**採用: リプライ型（最もシンプル）**

```
【元投稿（bot本体、既存機能）】
今日は、
・2008年07月02日　StatusNet（旧Laconica）のソースコードが公開
・2013年07月02日　Diasporaが分散化への取り組みを強化
だそうです！よかったね！

    ↓ リプライ（2-5分後）

【LLMコメント投稿】
わぁ〜！StatusNetって今のFediverseの
おじいちゃんみたいな存在なんだよね！
すごく大事な日だ〜✨ #分散SNS
```

**メリット**:
- 既存コードに影響なし
- LLM失敗時も通常投稿は成功
- タイムライン上で会話的に見える
- 将来の会話機能への布石

**実装上の注意**:
- 元投稿のnote IDを保存
- リプライAPIを使用（reply_id指定）

┌─────────────────────────────────────┐

│  ├─ システムプロンプト管理            │
│  ├─ コメント生成ロジック              │
│  └─ エラーハンドリング                │

#### 新規ファイル
- `llm_service.py` - LLM連携サービス
- `llm_prompts.py` - プロンプトテンプレート管理
- `config.py` - LLM関連設定追加

#### 設定項目（.env）
```bash
# LLM設定
LLM_ENABLED=false  # 初期はfalse、準備できたらtrue
LLM_API_URL=http://localhost:11434
LLM_MODEL=qwen3:1.7b-instruct-q4_K_M
LLM_TIMEOUT=30
LLM_MAX_TOKENS=100
LLM_TEMPERATURE=0.8  # やや高め（個性を出すため）
LLM_RETRY_COUNT=1  # リトライ回数

# コメント投稿設定
LLM_REPLY_DELAY=300  # 元投稿から何秒後にリプライするか（300秒=5分）
LLM_DRY_RUN=true  # テスト時はtrue（実際には投稿しない）

# パーソナリティ設定（Ollama導入後に調整）
LLM_PERSONALITY_FILE=prompts/default.txt  # システムプロンプトファイル
```

### 5. リスクと対応策

| リスク | 影響 | 対応策 |
|-------|------|--------|
| LLM応答遅延 | 投稿遅延 | タイムアウト設定（30秒）、失敗時は通常投稿 |
| メモリ不足 | サービス停止 | 軽量モデル使用、Swap設定確認 |
| 不適切な出力 | 評判リスク | システムプロンプトで制約、事後フィルタリング |
| Ollama停止 | 機能停止 | systemdでの自動再起動、フォールバック投稿 |

#### フォールバック戦略
- LLM応答失敗時は既存の投稿フォーマットで投稿
- エラーログを記録して後で確認
- 3回連続失敗時は一時的にLLM機能を無効化

## 🔧 実装の詳細設計

### データフロー詳細

```python
# main.pyまたは新規ファイル llm_scheduler.py

async def post_with_llm_commentary():
    # 1. 通常の定期投稿を実行（既存機能）
    note = await today_handler.handle_scheduled_post()
    note_id = note['createdNote']['id']
    
    # 2. LLM機能が有効かチェック
    if not config.llm_enabled:
        return
    
    # 3. 待機（設定された秒数）
    await asyncio.sleep(config.llm_reply_delay)
    
    # 4. LLMでコメント生成
    try:
        events_text = note['createdNote']['text']
        commentary = await llm_service.generate_commentary(events_text)
        
        # 5. リプライとして投稿
        await bot_client.create_note(
            text=commentary,
            reply_id=note_id,
            visibility='public'
        )
        logger.info(f"LLMコメント投稿成功: {len(commentary)}文字")
        
    except Exception as e:
        logger.error(f"LLMコメント生成失敗: {e}")
        # 失敗しても何もしない（既存投稿には影響なし）
```

### 試すべきバリエーション
- 一人称: わたし / ぼく / 自分の名前
- 語尾: 〜だよ / 〜なの / 〜だね / 〜です
- トーン: 元気 / のんびり / 技術的 / 文化的
- 絵文字: なし / 控えめ / 多め
- 専門性: オタク / 一般向け / バランス
